• extended JSON schema

        To describe the contents of resource, a JSON schema variant is used. This JSON schema is based on the JSON schema specification 04 and the JSON schema validation specification 00.
        The schema as used by the MEDvision360 APIs have the following extensions:

        validators
        Every propery (inside the 'properties' element) can have a 'validator' element. This element contains an array with javascript snippets used by the client to validate the value. 

        ...
"validator": [
  {
    "message": "Error message show to the user when the code returns false."
    "code": "return !!data['lastName'];"
  }
]
...

        enum titles
        An enum property can have an user friendly title to be shown in the user interface. These titles are stored in the 'options' item. 

        ...
"enum": [
  "FIRST",
  "SECOND"
],
"options": {
  "FIRST": {
    "title": "First"
  },
  "SECOND": {
    "title": "Second"
  }
}
...

      • tunnelled requests

        Currently most web browsers are having broken cache implementations in which they fail to see the difference between two resources which vary only by Content-Language and/or Content-Type.

        To deal with these buggy cache implementations, most resources are also available using a tunnelled URI. In this case the Content-Language and Content-Type are encoded in the URI and no headers are used.

        Because the URI looks different, the browser will fetch the correct version from the cache when switching between languages.

        For example, the resource /person/schema is also available as /person/schema.{lang}.json
        In this example the {lang} parameter needs to be replaced with the language code.

        These tunnelled resources will act in the same way as their normal counterparts except for one thing. The tunnelled resources do not return a 406 when the requested language does not exist, they return a 404 instead.
